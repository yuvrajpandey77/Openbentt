# Openbentt v2.2.4 — Compatibility hardening + privacy-safe growth metrics

This release focuses on two things:

1. Better desktop resilience on machines with unstable/missing GPU acceleration.
2. A practical, privacy-safe way to track product reach (site visits + installer downloads) without collecting personal user telemetry.

## What shipped

### 1) Desktop compatibility hardening

- Added one-time auto-recovery when the Electron GPU process crashes repeatedly during startup.
- On repeated GPU crash bursts, Openbentt relaunches once in software rendering mode (`OPENBENTT_DISABLE_GPU=1`).
- Existing Linux GPU safe-mode detection remains active for known-bad setups.

See implementation details in `electron/main.mjs` and `electron/gpuSafeMode.mjs`.

### 2) Setup + troubleshooting docs

- Added `docs/SETUP_GUIDE.md` with:
  - local web and desktop setup,
  - build/test checks,
  - Linux no-dGPU crash/lag troubleshooting and runtime overrides.

## New: how to track users safely (without invasive telemetry)

Openbentt already supports a privacy-first approach. For this release, the recommended measurement stack is:

### A) Site visitors (web traffic)

Use the existing **Vercel Analytics** toggle in app privacy settings.

- Metric: pageviews / visitors
- Where to view: Vercel project → **Analytics** dashboard
- Privacy model: aggregate web analytics, no Openbentt account required

### B) Desktop downloads

Use **GitHub Releases asset download counts**.

- Metric: per-installer download totals (AppImage, deb, exe, dmg)
- Where to view:
  - GitHub repository → Releases → latest tag → asset download numbers, or
  - GitHub API `releases/latest` for automated reporting

### C) Active users

By default, Openbentt does **not** send user-identifying telemetry.

- Out-of-the-box we do **not** claim exact DAU/MAU.
- If you need DAU/MAU later, add explicit opt-in telemetry with clear consent text and anonymized identifiers.

## Operator quick commands

```bash
# Desktop compatibility smoke (software fallback path)
OPENBENTT_DISABLE_GPU=1 npm run electron:start

# Standard validation before tagging
npm run lint
npm run test
npm run build
```

## Recommendation for production rollout

- Roll out in stages (canary first), then full rollout after monitoring:
  - crash reports / startup failures,
  - support tickets related to Linux GPU stacks,
  - web visit trend (Vercel Analytics),
  - release asset download trend (GitHub Releases).
