1. Executive Summary

Openbentt is a local-first, privacy-respecting multi-provider AI chat client packaged as both a static web app and an Electron desktop application. Users bring their own API keys (OpenRouter, OpenAI, Anthropic, Google Gemini, or any OpenAI-compatible endpoint) and run chats, document workspaces, research pipelines, and on-device language models — all without any backend account system.

Unique ValueWhy it matters
API keys never leave the user's browser or deviceNo server-side account, no data leakage risk
On-device inference (WebGPU/WASM + native GGUF)Works offline; privacy-critical use cases
Compare 2–4 models side-by-side with real timingInform model selection decisions with actual data
Document-grade workspaces (LaTeX ↔ AI ↔ PDF)Academic writing + AI in one surface
Research augmentation with citation graphsBibTeX, arXiv, Semantic Scholar, HF datasets
Current maturity

The core chat engine and workspace features are functionally complete. Key gaps: three workspace routes are hidden from nav, no integration/E2E tests, llama-server not bundled in releases, and macOS/Windows builds are unsigned.

3. What Openbentt Does — In Plain Language

3.1 Core Chat

Pick a provider and model, type a message, get a streamed reply. All threads stored in browser localStorage — no account, no server. Multiple threads with auto-named titles.

3.2 Model Comparison

Send one prompt → parallel replies from 2–4 models. Each reply shows Time to First Token, total response time, and token counts. Great for model evaluation on real tasks.

3.3 On-Device Inference

  • WebGPU / WASM (web + desktop): @huggingface/transformers runs Gemma, Qwen-0.5B, etc. inside the browser. Auto-falls back: WebGPU → WASM CPU → smaller model.
  • Local GGUF (desktop only): Download any GGUF from Hugging Face in the Labs panel. Electron spawns llama-server on localhost; the chat stack talks to it as an OpenAI-compatible API.

3.4 Notebook Workspace

Upload a PDF or write LaTeX. Ask AI to improve sections, add citations, fix errors. Changes diff before applying. Compiles PDF via BusyTeX WASM (in-browser) or remote pdflatex server.

3.5 Research Labs

  • BibTeX management and citation graph visualization
  • Hugging Face dataset card browser
  • Local GGUF Hub: search, download, manage, and chat with models
  • Research proxy: enriches queries with Wikipedia, arXiv, Semantic Scholar, Brave Search

3.6 Additional Workspaces

  • LaTeX Write (/write): KaTeX preview + PDF export — lighter than full Notebook
  • Benchmark (/benchmark): Run a prompt N times across models; plot latency distributions
  • WebGPU Diagnostics (/webgpu): Probe GPU, check adapter, run Gemma smoke test

3.7 Share, Export, Settings

Share any chat as a compressed URL (no server). Export to Markdown or PDF. Full settings: provider, model, theme, research toggle, GGUF path, HF token, keyboard shortcuts.

4. System Architecture

4.1 Three-Layer Model

Electron main process (Node) [desktop only]
app:// protocol → dist/ · BrowserWindow · IPC handlers · spawns llama-server (127.0.0.1) · HF download · userData registry · safeStorage HF token
Preload bridge (contextBridge)
Exposes window.openbenttLocalGguf and window.openbenttDesktop — no Node in renderer, context isolation, sandbox
Renderer — React SPA (same for web + desktop)
ChatContext → routes by aiProvider → OpenRouter (HTTPS) / WebGPU WASM (in-process) / Local GGUF (HTTP loopback via IPC) · localStorage keys + chat history

4.2 AI Provider Routing

ChatContext.runAssistantPipeline
 ├── "openrouter"        → streamOpenRouterChat (HTTPS to api.openrouter.ai)
 ├── "openai_direct"     → streamOpenRouterChat w/ openai base URL
 ├── "openai_compatible" → streamOpenRouterChat w/ custom base URL (Ollama, LM Studio…)
 ├── "webgpu_gemma"      → streamLocalGemmaChat (Transformers.js in-renderer)
 │    └── ensureLocalGemmaLoaded → WebGPU preferred, WASM fallback, tiny-model fallback
 └── "local_gguf"        → streamLocalGgufChat
      └── IPC: ensureServer(registryId) → main spawns llama-server
      └── HTTP POST 127.0.0.1:<port>/v1/chat/completions (OpenAI-compatible stream)

6. Current State Assessment

What works well

  • Core chat loop is solid; unit-tested parsers and helpers
  • Provider abstraction in aiStream.ts is clean and extensible
  • Notebook workflow is genuinely powerful (LaTeX ↔ AI ↔ PDF in one surface)
  • Electron shell is well-secured (context isolation, sandbox, no Node in renderer)
  • GGUF download / llama-server lifecycle is robust (resume download, disk check, SHA-256, cleanup on quit)
  • HF token handling uses OS-level encryption on desktop

Core concerns

Discoverability gap

Three full workspaces (/write, /benchmark, /webgpu) are invisible to users navigating via the sidebar. The product is larger than it appears from the nav.

Testing gap

ChatContext.tsx is the application's core state machine (~900 lines) and has zero test coverage. A provider or streaming change could silently break core chat.

Distribution gap

llama-server is the centerpiece of the desktop's local AI story but is not bundled in CI artifacts. Most desktop users hit a manual installation step.

Signing gap

macOS and Windows builds are unsigned. This is a blocking issue for non-technical users who see Gatekeeper / SmartScreen warnings.

Documentation drift

PRODUCTION_CHECKLIST.md and electron/README.md have stale references (preload.mjs vs actual preload.cjs; Electron first-load path note).

7. UX Improvement Plan

7.1 Navigation & Discovery

  • Add /write, /benchmark, /webgpu to WORKSPACE_NAV_ITEMS with appropriate icons (PenLine, BarChart2, Cpu)
  • Add a collapsible "Advanced" group in the sidebar if it becomes crowded
  • Add "What's new / Download latest" callout to Settings when a newer GitHub Release is detected

7.2 Setup & Onboarding

  • Rewrite the on-device setup card: remove hard GPU block; say "GPU recommended, CPU also works"
  • Show a hardware detection badge with the actual result (WebGPU / WASM / GPU model)
  • Desktop first-run: 3-step guided modal for GGUF ("Download a model → Pick Local GGUF → Chat"), gated by userData/onboarded.json

7.3 Chat Interface

  • True inline user message edit with confirmation + auto-regenerate (replaces composer-reload)
  • Retry history: keep previous responses behind "See previous" toggle (like ChatGPT retries)
  • Detect OpenAI tool-call JSON in assistant replies → render as collapsible "Tool call" cards
  • System prompt editor per thread (text field in thread settings sidebar)

7.4 Mobile

  • Use visualViewport resize handler to push chat input above keyboard
  • Swipe-to-switch-thread gesture on mobile viewport
  • Set overflow: hidden on body when soft keyboard is open to prevent scroll jank

7.5 Settings Refactor

  • Split single large panel into tabbed sections: General · Models · Providers · Desktop (Electron-only) · Privacy · Advanced
  • Desktop tab only visible when window.openbenttDesktop?.isElectron is true

7.6 Error States

  • Parse HTTP 429 x-ratelimit-reset-requests header → show "Try again in X sec" countdown
  • GGUF "binary not found" → show diagnostic checklist UI: ✅ Model downloaded · ❌ llama-server missing → [Install guide]
  • WebGPU load progress: show download % + estimated time for model weights
  • On API key missing: show actionable empty state with button → Settings (not just an error toast)

7.7 Performance

  • Add proper loading skeleton to LocalGgufHub while registry loads
  • Use startTransition for model selector updates to avoid blocking input
  • Verify Rollup manualChunks splits are actually reducing initial bundle size (measure with vite-bundle-analyzer)
  • Lazy-load heavy workspace deps (busytex, pdfjs, transformers) — partially done; validate with Lighthouse
